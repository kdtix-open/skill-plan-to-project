# Releasing plan-to-project

This repository uses a production-style release pattern built around:

- semantic versions in `pyproject.toml`
- a human-maintained `CHANGELOG.md`
- GitHub autogenerated release notes customized by `.github/release.yml`
- a tag-driven GitHub release workflow in `.github/workflows/release.yml`

## Issue intake and project tracking

New repo issues should come through the forms in `.github/ISSUE_TEMPLATE/`.
Each form applies a tracking label and routes the issue to Project V2 `kdtix-open/8`.

The issue-form `projects` key uses GitHub's `OWNER/PROJECT_NUMBER` format.
GitHub notes that the person opening the issue must have write access to the
project for that automatic routing to succeed. For broader contributor intake,
enable the project's built-in auto-add workflow as a project-side setting.

## Maintainer checklist

1. Make sure the work has landed on `main` with a clean PR history.
2. Confirm the PR or PRs are labeled for release note categorization:
   - `enhancement` for new features
   - `bug` for fixes
   - `documentation` for docs-only changes
3. Update `CHANGELOG.md`:
   - move the relevant entries from `## [Unreleased]` into a new version section
   - capture any operator-facing workarounds or upgrade notes that autogenerated release notes will not say clearly
4. Bump the version in `pyproject.toml`.
5. Commit the version and changelog changes.
6. Tag the release and push it:

```bash
git checkout main
git pull --ff-only
git tag vX.Y.Z
git push origin main
git push origin vX.Y.Z
```

## What the release workflow does

When a `v*` tag is pushed, GitHub Actions runs the release workflow and calls:

```bash
gh release create vX.Y.Z --title "vX.Y.Z" --generate-notes --latest
```

That creates the GitHub Release and uses autogenerated release notes. The output
is shaped by `.github/release.yml`, which groups labeled pull requests into
feature, fix, and documentation sections.

## After publishing

- Review the GitHub Release page to confirm the generated notes look right.
- If a release needs extra operator guidance, edit the release body and add the
  missing workaround or migration note manually.
- Keep `CHANGELOG.md` as the durable, repo-local source of human-authored notes.
